# 요구사항 문서

## 소개

소설 작성 기록 관리를 위한 개인용 REST API 서버이다. 사용자는 여러 편의 소설을 관리하며, 각 소설의 챕터와 내용을 클라이언트 프로그램을 통해 서버의 데이터베이스에 저장하고 조회할 수 있다. 데이터베이스에 직접 접근하지 않고 REST API를 통해서만 데이터를 관리한다. 개인용 서버이므로 대규모 트래픽은 고려하지 않으며, 클라이언트는 어디서든 접속 가능한 형태로 제공된다.

## 용어 정의

- **API_서버**: 소설 데이터를 관리하는 REST API 서버 애플리케이션
- **클라이언트**: API_서버에 HTTP 요청을 보내 소설 데이터를 관리하는 프로그램
- **소설**: 제목, 설명, 생성일, 수정일 등의 메타데이터를 가지는 작품 단위
- **챕터**: 소설에 속하는 개별 장으로, 제목, 본문, 순서, 생성일, 수정일 등의 정보를 가짐
- **메타데이터**: 소설 또는 챕터의 제목, 설명, 날짜 등 본문 외의 부가 정보
- **인증_토큰**: 클라이언트가 API_서버에 요청 시 본인 확인을 위해 HTTP 헤더에 포함하는 문자열 값
- **인증되지_않은_요청**: 유효한 인증_토큰이 포함되지 않았거나 잘못된 인증_토큰이 포함된 HTTP 요청

## 요구사항

### 요구사항 1: 소설 생성

**사용자 스토리:** 작가로서, 새로운 소설을 등록하고 싶다. 그래야 작품별로 기록을 분리하여 관리할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 정보(제목, 설명)를 포함한 생성 요청을 수신하면, THE API_서버 SHALL 새로운 소설을 데이터베이스에 저장하고 생성된 소설 정보를 응답으로 반환한다
2. WHEN 소설 생성 요청에 제목이 누락되면, THE API_서버 SHALL 400 상태 코드와 함께 누락된 필드를 명시하는 오류 메시지를 반환한다
3. WHEN 소설이 성공적으로 생성되면, THE API_서버 SHALL 생성일과 수정일을 자동으로 기록한다

### 요구사항 2: 소설 목록 조회

**사용자 스토리:** 작가로서, 등록된 모든 소설의 목록을 조회하고 싶다. 그래야 작업할 소설을 선택할 수 있다.

#### 인수 조건

1. WHEN 소설 목록 조회 요청을 수신하면, THE API_서버 SHALL 등록된 모든 소설의 메타데이터 목록을 응답으로 반환한다
2. WHEN 등록된 소설이 없으면, THE API_서버 SHALL 빈 목록을 응답으로 반환한다

### 요구사항 3: 소설 상세 조회

**사용자 스토리:** 작가로서, 특정 소설의 상세 정보를 조회하고 싶다. 그래야 소설의 현재 상태를 파악할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자를 포함한 상세 조회 요청을 수신하면, THE API_서버 SHALL 해당 소설의 메타데이터와 챕터 목록을 응답으로 반환한다
2. IF 존재하지 않는 소설 식별자로 조회 요청을 수신하면, THEN THE API_서버 SHALL 404 상태 코드와 함께 소설을 찾을 수 없다는 오류 메시지를 반환한다

### 요구사항 4: 소설 정보 수정

**사용자 스토리:** 작가로서, 소설의 제목이나 설명을 수정하고 싶다. 그래야 작품 정보를 최신 상태로 유지할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자와 수정할 정보를 포함한 수정 요청을 수신하면, THE API_서버 SHALL 해당 소설의 정보를 갱신하고 수정된 소설 정보를 응답으로 반환한다
2. WHEN 소설 정보가 성공적으로 수정되면, THE API_서버 SHALL 수정일을 자동으로 갱신한다
3. IF 존재하지 않는 소설 식별자로 수정 요청을 수신하면, THEN THE API_서버 SHALL 404 상태 코드와 함께 소설을 찾을 수 없다는 오류 메시지를 반환한다

### 요구사항 5: 소설 삭제

**사용자 스토리:** 작가로서, 더 이상 필요 없는 소설을 삭제하고 싶다. 그래야 관리 목록을 깔끔하게 유지할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자를 포함한 삭제 요청을 수신하면, THE API_서버 SHALL 해당 소설과 소설에 속한 모든 챕터를 데이터베이스에서 삭제하고 성공 응답을 반환한다
2. IF 존재하지 않는 소설 식별자로 삭제 요청을 수신하면, THEN THE API_서버 SHALL 404 상태 코드와 함께 소설을 찾을 수 없다는 오류 메시지를 반환한다

### 요구사항 6: 챕터 생성

**사용자 스토리:** 작가로서, 소설에 새로운 챕터를 추가하고 싶다. 그래야 소설의 내용을 장 단위로 작성할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자와 챕터 정보(제목, 본문)를 포함한 생성 요청을 수신하면, THE API_서버 SHALL 해당 소설에 새로운 챕터를 추가하고 생성된 챕터 정보를 응답으로 반환한다
2. WHEN 챕터 생성 요청에 제목이 누락되면, THE API_서버 SHALL 400 상태 코드와 함께 누락된 필드를 명시하는 오류 메시지를 반환한다
3. WHEN 챕터가 성공적으로 생성되면, THE API_서버 SHALL 챕터의 순서 번호를 자동으로 부여하고 생성일과 수정일을 기록한다
4. IF 존재하지 않는 소설 식별자로 챕터 생성 요청을 수신하면, THEN THE API_서버 SHALL 404 상태 코드와 함께 소설을 찾을 수 없다는 오류 메시지를 반환한다

### 요구사항 7: 챕터 상세 조회

**사용자 스토리:** 작가로서, 특정 챕터의 내용을 조회하고 싶다. 그래야 이전에 작성한 내용을 확인하고 이어서 작성할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자와 챕터 식별자를 포함한 조회 요청을 수신하면, THE API_서버 SHALL 해당 챕터의 제목, 본문, 순서, 생성일, 수정일을 응답으로 반환한다
2. IF 존재하지 않는 소설 식별자 또는 챕터 식별자로 조회 요청을 수신하면, THEN THE API_서버 SHALL 404 상태 코드와 함께 해당 리소스를 찾을 수 없다는 오류 메시지를 반환한다

### 요구사항 8: 챕터 내용 수정

**사용자 스토리:** 작가로서, 챕터의 제목이나 본문을 수정하고 싶다. 그래야 작성 중인 소설의 내용을 다듬을 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자, 챕터 식별자, 수정할 정보를 포함한 수정 요청을 수신하면, THE API_서버 SHALL 해당 챕터의 정보를 갱신하고 수정된 챕터 정보를 응답으로 반환한다
2. WHEN 챕터 정보가 성공적으로 수정되면, THE API_서버 SHALL 수정일을 자동으로 갱신한다
3. IF 존재하지 않는 소설 식별자 또는 챕터 식별자로 수정 요청을 수신하면, THEN THE API_서버 SHALL 404 상태 코드와 함께 해당 리소스를 찾을 수 없다는 오류 메시지를 반환한다

### 요구사항 9: 챕터 삭제

**사용자 스토리:** 작가로서, 불필요한 챕터를 삭제하고 싶다. 그래야 소설의 구성을 정리할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자와 챕터 식별자를 포함한 삭제 요청을 수신하면, THE API_서버 SHALL 해당 챕터를 데이터베이스에서 삭제하고 성공 응답을 반환한다
2. WHEN 챕터가 삭제되면, THE API_서버 SHALL 나머지 챕터의 순서 번호를 자동으로 재정렬한다
3. IF 존재하지 않는 소설 식별자 또는 챕터 식별자로 삭제 요청을 수신하면, THEN THE API_서버 SHALL 404 상태 코드와 함께 해당 리소스를 찾을 수 없다는 오류 메시지를 반환한다

### 요구사항 10: 챕터 순서 변경

**사용자 스토리:** 작가로서, 챕터의 순서를 변경하고 싶다. 그래야 소설의 구성을 자유롭게 재배치할 수 있다.

#### 인수 조건

1. WHEN 유효한 소설 식별자와 챕터 순서 목록을 포함한 순서 변경 요청을 수신하면, THE API_서버 SHALL 챕터의 순서를 요청된 대로 갱신하고 갱신된 챕터 목록을 응답으로 반환한다
2. IF 순서 변경 요청에 포함된 챕터 식별자가 해당 소설에 속하지 않으면, THEN THE API_서버 SHALL 400 상태 코드와 함께 유효하지 않은 챕터 식별자를 명시하는 오류 메시지를 반환한다

### 요구사항 11: REST API 설계 원칙

**사용자 스토리:** 클라이언트 개발자로서, 일관된 REST API 규칙을 따르는 서버를 사용하고 싶다. 그래야 클라이언트 개발이 용이하다.

#### 인수 조건

1. THE API_서버 SHALL 모든 요청과 응답에 JSON 형식을 사용한다
2. THE API_서버 SHALL RESTful 리소스 경로 규칙을 따른다 (소설: /novels, 챕터: /novels/{novelId}/chapters)
3. THE API_서버 SHALL 성공 응답에 적절한 HTTP 상태 코드를 사용한다 (생성: 201, 조회: 200, 수정: 200, 삭제: 204)
4. IF 요청 본문이 유효한 JSON 형식이 아니면, THEN THE API_서버 SHALL 400 상태 코드와 함께 형식 오류 메시지를 반환한다

### 요구사항 12: 데이터 영속성

**사용자 스토리:** 작가로서, 서버를 재시작해도 데이터가 유지되길 원한다. 그래야 작성한 소설 데이터를 안전하게 보관할 수 있다.

#### 인수 조건

1. THE API_서버 SHALL 모든 소설과 챕터 데이터를 영구 저장소(데이터베이스)에 저장한다
2. WHEN API_서버가 재시작되면, THE API_서버 SHALL 이전에 저장된 모든 데이터를 정상적으로 제공한다

### 요구사항 13: 인증 토큰 기반 사용자 인증

**사용자 스토리:** 작가로서, 인증된 사용자만 API에 접근할 수 있도록 하고 싶다. 그래야 허가되지 않은 사람이 소설 데이터를 열람하거나 수정하는 것을 방지할 수 있다.

#### 인수 조건

1. THE API_서버 SHALL 모든 API 요청에 대해 HTTP 헤더에 포함된 인증_토큰을 검증한다
2. WHEN 유효한 인증_토큰이 포함된 요청을 수신하면, THE API_서버 SHALL 해당 요청을 정상적으로 처리한다
3. IF 인증되지_않은_요청을 수신하면, THEN THE API_서버 SHALL 401 상태 코드와 함께 인증이 필요하다는 오류 메시지를 반환한다
4. IF 인증_토큰이 요청 헤더에 누락되면, THEN THE API_서버 SHALL 401 상태 코드와 함께 인증_토큰이 필요하다는 오류 메시지를 반환한다
5. THE API_서버 SHALL 인증_토큰을 서버 환경 변수로 설정할 수 있도록 지원한다
